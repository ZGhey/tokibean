# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Tokibean 码豆 ("拱门·墩墩"): a Tauri 2 desktop pet that monitors Claude Code activity (via hooks) and token usage (via local JSONL logs, or the official Anthropic usage API). Transparent, always-on-top, frameless window with a tray icon. Cross-platform (Windows/macOS/Linux), though several features (WSL sync, Windows Credential Manager, win32 window-focus FFI) are Windows-specific. Code comments and developer logs are in English. In-app UI text is **bilingual**: wrap user-facing strings with `i18n::t("中文", "English")` (backend, `src-tauri/src/i18n.rs`) or the `t("key")` dictionary (frontend, `src/main.js`) so the UI follows the system language — Chinese for `zh*` locales, English otherwise. Exception: the pet's tool-animation status keys (`cmd`/`reading`/`coding`/…, from `friendly_tool`) are stable English keys the renderers match on — never localize them. The README is English-primary (`README.md`) with a Chinese translation kept in sync at `README.zh-CN.md`; update both when either changes.

## Commands

```bash
npm install        # one-time; installs @tauri-apps/cli
npm run dev        # tauri dev — compiles Rust and launches with hot-reloaded frontend
npm run build      # tauri build — production bundles (macOS: run `npm run tauri icon app-icon.png` first for icns)
```

Rust-only checks (faster than a full dev launch): `cargo check` / `cargo clippy` inside `src-tauri/`.

There are no tests and no linter configured. The frontend is plain HTML/CSS/JS with no build step — `src/` is served directly (`frontendDist: "../src"` in tauri.conf.json), so frontend edits need no compilation, just a reload (`npm run dev` hot-reloads it).

CI (`.github/workflows/build.yml`) builds macOS (universal) and Windows bundles via `workflow_dispatch` or on `v*` tags (tags also cut a GitHub Release). Linux isn't built in CI.

## Architecture

Two halves connected by Tauri events and commands:

**Rust backend (`src-tauri/src/`)** — owns all state, runs three background threads spawned in `main.rs`'s `setup`:

1. **Hook server thread** (`hooks_server.rs`): a `tiny_http` server on `127.0.0.1:8737` (configurable via `cfg.bind`/`cfg.port`). Claude Code hooks POST event JSON to `/event`; `hook_event_name` drives per-session state transitions (`UserPromptSubmit`/`PreToolUse`→Working, `PostToolUse` error→4s "oops" flag, `Stop`→Done + bubble with duration/celebrate level + today's stop counter, `Notification`→Attention, `SessionStart`→"开工!" bubble if nobody else is busy, `SessionEnd`→drop the session). State is per `session_id` (HashMap in `Core`), aggregated for display: any attention > any working > any done > limit > idle. `run_in_background` tool calls push a 15-minute "satellite" bg-task marker (hooks have no completion event for background tasks, so it just decays). CAUTION: `push_update` locks `core` — never call it while already holding the `core` lock (past deadlock).
2. **Heartbeat thread** (`main.rs`): every 1s expires transients (Done states, bubbles, tool notes, oops, stale sessions >6h, bg tasks) via `state::expire_transients`; every 30s rescans usage JSONL (`state::refresh_usage`) and checks 80%/100% alerts (`state::check_usage_alerts`); also throttled window-position save as a backstop. Official usage API fetches are event-driven, decided inside `refresh_usage`: on startup/mode-switch/account-connect (`with_official=true`), on Stop/panel-open (via the `official_want` flag), while working with a >5min-old cache, or once the cached reset time passes — never while fully idle. All triggers share a 60s debounce (`official_last_try`), a 5min rate-limit backoff (`official_backoff`), and a 15min token-refresh backoff (`refresh_backoff` — without it an expired credential would hammer the token endpoint into rate-limiting the user's IP). Cached official values are kept up to 6h on fetch failure before falling back to local estimation.
3. **Click-through thread** (`main.rs`): every 50ms, toggles `set_ignore_cursor_events` based on cursor position — clicks pass through transparent empty window area to whatever's beneath, except over the pet/bubble canvas region or an expanded panel (height > 345 logical px means the whole window is solid; otherwise only the bottom canvas strip is).

Both threads (plus command handlers) mutate `state::Shared` (Mutexes + atomics, managed by Tauri) and push a full `PetUpdate` snapshot to the frontend via `app.emit("pet-update", ...)`.

- `state.rs` — the state machine. Base states with priority: attention > working > done > limit > idle. `limit` (sleeping) is derived, not stored: it's `idle` while subscription block usage ≥ 100%, but only when `pct_valid` (basis is `official` or `manual` — the `auto` local estimate is display-only and would false-trigger "100%" whenever it sets a new personal record). `warn` (≥ 80%) is an overlay flag, not a state. `check_usage_alerts` also force-idles any Working/Attention session when usage hits 100% (limit means no Stop event will ever arrive to end it) and posts an explanatory bubble.
- `usage.rs` — incremental parser of `~/.claude/projects/**/*.jsonl` (plus, on Windows, every WSL distro's `~/.claude/projects` via `\\wsl$`, rediscovered every 10 min). `Scanner` remembers per-file byte offsets and only reads appended, newline-complete data; dedupes by `message.id:requestId` (session resume duplicates lines across files); keeps 8 days of events. `build_snapshot` groups events into 5-hour billing blocks (start floored to the UTC hour of first activity — matches ccusage semantics). The block-usage percentage is either `manual` (token count vs. configured `block_limit`) or `auto` (weighted-cost vs. the historical max-cost block — raw token counts would be skewed by cheap cache-read tokens dominating the denominator); `official` percentage (from the Anthropic usage API) overrides both when available.
- `config.rs` — `~/.config/claude-pet/config.json` (`%APPDATA%\claude-pet` on Windows). `resolved_mode()` decides subscription vs API: explicit setting, else heuristic (`ANTHROPIC_API_KEY` present → api).
- `official.rs` — fetches `https://api.anthropic.com/api/oauth/usage` via `curl` (token passed over curl's stdin with `-H @-`, never on the command line/process list). `get_token` does a three-pass scan for a usable Claude Code credential: pass 1, anything already valid with zero network calls (pet's own stored OAuth token, local `~/.claude/.credentials.json`, macOS Keychain via `security find-generic-password -s "Claude Code-credentials"`, WSL distros' `.credentials.json` read-only); pass 2, refresh an expired token if the 15-min refresh backoff allows it (pet token, then `.credentials.json`, writing the refreshed token back to disk in Claude Code's own format); pass 3, fallback stores — Windows Credential Manager (hand-written `CredReadW` FFI), then a manually configured long-lived token / `CLAUDE_CODE_OAUTH_TOKEN` env var. Credentials pulled from any credential store are still expiry-checked before use since they can't be refreshed in place. macOS Keychain is checked in pass 1, not pass 3: on macOS the CLI keeps the Keychain entry fresh but may leave `~/.claude/.credentials.json` stale/vestigial (its refresh token can go dead), so `.credentials.json`'s refresh attempt in pass 2 must never be reached ahead of a perfectly valid Keychain token — otherwise the pet keeps retrying a dead refresh against `console.anthropic.com/v1/oauth/token`, which can get the endpoint rate-limited for the manual "连接 Claude 账号" flow (`login.rs`) too, since it's the same server-side endpoint.
- `login.rs` — in-app "connect Claude account" OAuth PKCE flow: opens the system browser to `claude.ai/oauth/authorize`, a local `tiny_http` server on port 54545 catches the `/callback`, then exchanges the code for tokens via `console.anthropic.com/v1/oauth/token` and stores them in config. 3-minute timeout; validates the OAuth `state` param.
- `focus.rs` — "click the bubble to focus the terminal" (`focus_terminal` command). macOS: `osascript ... activate` tried against a fixed app-name list (`Claude`, `iTerm2`, `Terminal`, `Visual Studio Code`). Windows: hand-written Win32 FFI (`EnumWindows`/`GetWindowThreadProcessId`/`QueryFullProcessImageNameW`) ranks visible top-level windows by a process-name allowlist, then by Z-order, and calls `SetForegroundWindow`. There's no session→window mapping from hooks, so this is always a heuristic "most plausible terminal", not a precise match. Linux: unimplemented (returns an error).
- `hooks_install.rs` — merges 7 curl-command hooks into `~/.claude/settings.json` (backs up to `settings.json.bak-claude-pet` first; idempotent by checking for the `:<port>/event` marker string in each event's hook array; `incomplete()` reports if any event or WSL distro is missing the marker, so the panel can re-surface the install button after an upgrade). On Windows it also syncs hooks into every WSL distro's existing `~/.claude` (via `\\wsl$`) — Claude Code in WSL reads the Linux-side settings, so Windows-only install is invisible to it. Mirrored networking uses `127.0.0.1`; NAT mode embeds `$(ip route show default | awk '{print $3}')` as host and additionally needs `bind: "0.0.0.0"` in the pet config. Uses curl commands rather than http-type hooks for Claude Code version compatibility.

**Frontend (`src/`)** — no framework, uses `window.__TAURI__` global (`withGlobalTauri: true`):

- `main.js` — listens for `pet-update`, invokes Tauri commands (`get_update`, `install_hooks`, `set_mode`, `get_config`, `set_config`, `connect_claude`, `focus_terminal`, `panel_opened`), renders the usage panel, runs the canvas animation loop. Also owns: window drag-to-move (native `startDragging`, distinguished from a click by a 4px movement threshold), the expand/collapse panel animation, an 8-bit WebAudio beep on state changes, and dynamic skin loading (injects `skins/<name>.js` as a `<script>` tag when a non-default skin is configured).
- `pet.js` — the skin layer, deliberately isolated. All drawing lives here behind one interface: `window.PetRenderer.draw(ctx, canvas, state, warn, bubble, t, extra)` with states `idle | working | attention | done | limit`. `extra` is optional (`{sessions, workSecs, attnSecs, toolNote, celebrate, oops, bgCount, dragging, pat}`) — old skins that ignore it must keep working. Replacing this file swaps the skin; do not leak rendering logic into main.js.
- Skins live in `src/skins/*.js` (override `window.PetRenderer`, may reuse `window.PetKit`). The default skin in pet.js IS the original mascot "拱门·墩墩" (persimmon arch); `bean.js`/`tabby.js` are the other built-in skins. `src/skins/tribute.js`, if present, is a LOCAL-ONLY skin (gitignored) — never commit it or recreate its likeness in the repo.
- The panel expands the window upward on toggle (`setWindowHeight` in main.js keeps the bottom edge fixed and always measures from the window's *current* height — do not assume it's at `BASE_H` (340)). Window position persists to config (throttled save on Moved, skipped within 1.2s of a programmatic resize, since panel toggle moves-then-resizes and that "move" isn't a user drag).
- `gen-icon.js` — zero-dependency Node script that hand-draws `app-icon.png` (1024×1024) pixel-by-pixel and PNG-encodes via built-in `zlib`, mirroring the default skin's pixel geometry. Run before `npm run tauri icon app-icon.png` if the mascot art changes.

## Constraints

- The default skin references the official pixel mascot and is for personal use only — if asked to prepare this project for public distribution, `pet.js` needs an original character and "Claude" must be dropped from the name (see README).
- Changing the port requires reinstalling hooks (the port is baked into the curl commands in settings.json).
- Rust holds locks briefly and clones snapshots out; keep that pattern — the hook server, heartbeat, and click-through threads share every Mutex in `Shared`.
- OAuth/credential handling: tokens must never appear in a process's command-line args (always pipe via curl stdin) and never get logged; credential-store reads should stay read-only where a refresh could kick another client (e.g. WSL's own Claude Code CLI) offline.
