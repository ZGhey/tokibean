# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Tokibean 码豆 ("拱门·墩墩"): a Tauri 2 desktop pet that watches your coding agents — **Claude Code, Codex, and Hermes Agent** — via hooks, and their quota/token usage (via each agent's local logs — Claude/Codex JSONL, Hermes SQLite `state.db` — plus the official Anthropic usage API for Claude). One pet aggregates every agent and session; each agent keeps its own quota window (Claude's 5-hour block, Codex's own — never normalized together; Hermes has no quota basis, only token counts). Hermes is **multi-profile**: every profile (`~/.hermes` default plus `~/.hermes/profiles/<name>/`) is discovered, hooked, and scanned; sessions are keyed `(agent, profile, session_id)`. Transparent, always-on-top, frameless window with a tray icon. Cross-platform (Windows/macOS/Linux), though several features (WSL sync, Windows Credential Manager) are Windows-specific. Code comments and developer logs are in English. In-app UI text is **bilingual**: wrap user-facing strings with `i18n::t("中文", "English")` (backend, `src-tauri/src/i18n.rs`) or the `t("key")` dictionary (frontend, `src/main.js`) so the UI follows the system language — Chinese for `zh*` locales, English otherwise. Exception: the pet's tool-animation status keys (`cmd`/`reading`/`coding`/…, from `friendly_tool`) are stable English keys the renderers match on — never localize them. The README is English-primary (`README.md`) with a Chinese translation kept in sync at `README.zh-CN.md`; update both when either changes.

## Commands

```bash
npm install        # one-time; installs @tauri-apps/cli
npm run dev        # tauri dev — compiles Rust and launches with hot-reloaded frontend
npm run build      # tauri build — production bundles (macOS: run `npm run tauri icon app-icon.png` first for icns)
```

Rust-only checks (faster than a full dev launch): `cargo check` / `cargo clippy` inside `src-tauri/`.

The Rust side has a real test suite (`cargo test` in `src-tauri/`, 116 tests) — the reducer, the usage/codex parsers, the hook installers, and the config adoption are all covered, and new logic there is expected to be. No linter is configured. The frontend is plain HTML/CSS/JS with no build step — `src/` is served directly (`frontendDist: "../src"` in tauri.conf.json), so frontend edits need no compilation, just a reload (`npm run dev` hot-reloads it).

CI (`.github/workflows/build.yml`) builds macOS (universal) and Windows bundles via `workflow_dispatch` or on `v*` tags (tags also cut a GitHub Release). Linux isn't built in CI.

## Architecture

Two halves connected by Tauri events and commands:

**Rust backend (`src-tauri/src/`)** — owns all state, runs three background threads spawned in `main.rs`'s `setup`:

1. **Hook server thread** (`hooks_server.rs`): a `tiny_http` server on `127.0.0.1:8737` (configurable via `cfg.bind`/`cfg.port`). Agent hooks POST event JSON to `/event` (Claude — bare path, so pre-multi-agent installs keep working), `/event/codex`, or `/event/hermes?profile=<name>` — the agent slug is decided at install time and carried in the URL path, never sniffed from the payload; the Hermes profile rides the query param. `hook_event_name` drives per-session state transitions (`UserPromptSubmit`/`PreToolUse`→Working, `PostToolUse` error→4s "oops" flag, `Stop`→Done + bubble with duration/celebrate level + today's stop counter, `Notification`→Attention, `SessionStart`→"开工!" bubble if nobody else is busy, `SessionEnd`→drop the session). State is per `SessionKey` (`agent`, `profile`, `session_id` — profile is empty for Claude/Codex; HashMap in `Core`), aggregated for display: any attention > any working > any done > limit > idle. `run_in_background` tool calls push a 15-minute "satellite" bg-task marker (hooks have no completion event for background tasks, so it just decays). CAUTION: `push_update` locks `core` — never call it while already holding the `core` lock (past deadlock).
2. **Heartbeat thread** (`main.rs`): every 1s expires transients (Done states, bubbles, tool notes, oops, stale sessions >6h, bg tasks) via `state::expire_transients`; every 30s rescans usage JSONL (`state::refresh_usage`) and checks 80%/100% alerts (`state::check_usage_alerts`); also throttled window-position save as a backstop. Official usage API fetches follow Claude's actions (like Claude Code itself — no wall-clock polling), decided inside `refresh_usage`: on startup/mode-switch/account-connect (`with_official=true`), on the `official_want` flag raised by a hook (PreToolUse activity / Stop completion, set in `hooks_server.rs`) or panel-open, or a one-shot correction once the cached reset time passes — never on a timer, never while idle. The flag-driven fetches share a 60s debounce (`official_last_try`) so a tool-heavy turn caps at ~1 fetch/min; also a 5min usage-endpoint rate-limit backoff (`official_backoff`) and a token-refresh backoff (`refresh_backoff`): a plain refresh failure backs off 15min, but a *rate-limited* refresh backs off **6h**. The live token endpoint is `platform.claude.com/v1/oauth/token` (the old `console.anthropic.com/v1/oauth/token` now 404s), called with a neutral `User-Agent: axios/1.7.9` — Anthropic rate-limits token requests whose UA looks like Claude Code's, so a neutral UA sidesteps the throttle entirely (this, not IP/frequency, was the real cause of the old "接口限流" wall). The 6h refresh backoff is a safety net for the rare case a refresh *is* rate-limited: the token endpoint's lockout runs ~6h with no `Retry-After` and each attempt inside it re-arms the clock, so the backoff must exceed the lockout or a shorter retry keeps it locked forever. (Anthropic OAuth facts, from research: access token ~8h; refresh token rotates on every use and invalidates the old one — which is exactly why the pet uses only its own connected credential and never shares/borrows the CLI's, since two clients refreshing one credential would `invalid_grant` each other.) Cached official values are kept up to 6h on fetch failure; with no cache and no connected account there is simply no 5-hour-window percentage (the local estimate was removed — see `usage.rs`).
3. **Click-through thread** (`main.rs`): every 50ms, toggles `set_ignore_cursor_events` based on cursor position — clicks pass through transparent empty window area to whatever's beneath, except over the pet/bubble canvas region or an expanded panel (height > 345 logical px means the whole window is solid; otherwise only the bottom canvas strip is).

Both threads (plus command handlers) mutate `state::Shared` (Mutexes + atomics, managed by Tauri) and push a full `PetUpdate` snapshot to the frontend via `app.emit("pet-update", ...)`.

- `state.rs` — the state machine. Base states with priority: attention > working > done > limit > idle. `limit` (sleeping) is derived, not stored: it's `idle` while subscription block usage ≥ 100%, but only when `pct_valid` (basis is `official` or `manual`; when neither is available basis is `none` and there's no percentage at all — the old `auto` weighted-cost estimate was removed, as it was display-only and would false-trigger "100%" whenever it set a new personal record). `warn` (≥ 80%) is an overlay flag, not a state. `check_usage_alerts` also force-idles any Working/Attention session when usage hits 100% (limit means no Stop event will ever arrive to end it) and posts an explanatory bubble.
- `usage.rs` — incremental parser of `~/.claude/projects/**/*.jsonl` (plus, on Windows, every WSL distro's `~/.claude/projects` via `\\wsl$`, rediscovered every 10 min). `Scanner` remembers per-file byte offsets and only reads appended, newline-complete data; dedupes by `message.id:requestId` (session resume duplicates lines across files); keeps 8 days of events. `build_snapshot` groups events into 5-hour billing blocks (start floored to the UTC hour of first activity — matches ccusage semantics). The block-usage percentage is `manual` (token count vs. configured `block_limit`) when a limit is set, else basis `none` (no percentage); `official` percentage (from the Anthropic usage API) overrides it when available. An earlier `auto` weighted-cost estimate (vs. the historical max-cost block) was **removed** — without a real quota basis it was too speculative to show as a percentage, so the pet now shows no window percentage until you connect an account or set a manual `block_limit`.
- `config.rs` — `~/.config/tokibean/config.json` (macOS: `~/Library/Application Support/tokibean/`; Windows: `%APPDATA%\tokibean`). The directory was `claude-pet` before 0.5.0; `Config::load` **adopts** that file once if ours is absent, copying it (never moving it) to the new path — the pure `choose()` decides, and our own file always wins so a stale `claude-pet` copy can't resurrect an already-rotated refresh token. Never delete the old directory (ADR-0015). `resolved_mode()` decides subscription vs API: explicit setting, else heuristic (`ANTHROPIC_API_KEY` present → api).
- `official.rs` — fetches `https://api.anthropic.com/api/oauth/usage` via `curl` (token passed over curl's stdin with `-H @-`, never on the command line/process list). `get_token` uses **only the pet's own connected credential** — the one the user obtains once via the panel's "连接 Claude 账号" (`login.rs`), stored in the pet's own config (`oauth_access`/`oauth_refresh`/`oauth_expires_ms`) and refreshed independently against `platform.claude.com/v1/oauth/token` (neutral `axios` UA to dodge the Claude-Code-UA rate limit): valid → use as-is; expired → refresh with its own refresh token (rotated token written back to config), backing off 15min on transient failure / 6h on rate-limit. A dead refresh token (`invalid_grant`) wipes the stored credential and raises `reconnect_needed`, so the panel reverts to "未连接" and re-shows the connect button. As a standalone app it deliberately does **not** borrow another client's token (local `~/.claude/.credentials.json`, WSL distros, macOS Keychain, Windows Credential Manager) — those rotate/expire/rate-limit outside our control and would drag the pet down with them (chronic rate-limit + kicking the WSL CLI offline). The only other, explicitly-opt-in source is a manually configured long-lived token (`cfg.oauth_token`) / `CLAUDE_CODE_OAUTH_TOKEN` env var. Consequence: official usage requires connecting once; before that the panel shows "未连接" and there is no 5-hour-window percentage (local estimation was removed).
- `login.rs` — in-app "connect Claude account" OAuth PKCE flow: opens the system browser to `claude.ai/oauth/authorize`, a local `tiny_http` server on port 54545 catches the `/callback`, then exchanges the code for tokens via `platform.claude.com/v1/oauth/token` (neutral `axios` UA; the exchange body must include the `state` field) and stores them in config. 3-minute timeout; validates the OAuth `state` param. A rate-limited exchange arms a 6h `connect_cooldown` so repeat clicks can't keep re-arming the endpoint's lockout.
- `hermes_install.rs` + `plugin_hermes.py`/`plugin_hermes.yaml` — Hermes Agent integration, an **in-process Python plugin** (not shell hooks — an earlier shell-hook bridge was replaced; `install` still cleans up its leftovers: old `hooks`/`hooks_auto_accept` config blocks and the `agent-hooks/tokibean-bridge.py` script). `profile_configs()` finds every profile config: `~/.hermes/config.yaml` (name "default") plus each `~/.hermes/profiles/<name>/config.yaml`; `HERMES_HOME` is honored, including when it points *inside* a profile directory (walks up two levels). Install writes the plugin into **every profile's own** `plugins/tokibean/` (Hermes discovers plugins per-profile home, not globally — the global `~/.hermes/plugins/` copy is only for the default profile) and adds `tokibean` to `plugins.enabled` in each config.yaml (backup `config.yaml.bak-tokibean`, idempotent). The plugin maps Hermes lifecycle hooks (`pre_tool_call`/`post_llm_call`/`pre_approval_request`/…) onto Claude-style `hook_event_name`s and POSTs to `/event/hermes?profile=<name>`, detecting its profile from `HERMES_HOME`.
- `hermes_usage.rs` — token scanner over every profile's `state.db` (SQLite: `sessions` table, `tui`/`cli` sources, input+output tokens). Scanned on the 30s heartbeat; today's total and 7-day dailies are folded into the panel's cross-agent totals (`daily_hermes` carries the per-day Hermes share for hover). No quota percentage — Hermes quota basis is `none`, tokens only.
- `agents.rs` — per-agent presence/site resolution: manual dir from Settings > env var (`CLAUDE_CONFIG_DIR`/`CODEX_HOME`/`HERMES_HOME` — a free hint only; a Dock-launched GUI never sees your shell env) > default `~/.claude`/`~/.codex`/`~/.hermes`. "Installed" = the directory exists. Hermes gets one Settings site per profile (no generic site); Claude on Windows also gets one site per running WSL distro.
- `codex_install.rs` — merges 7 curl-command hooks into `~/.codex/hooks.json` (`/event/codex` path), stripping the copies Codex's importer made of our Claude hooks. Codex won't RUN a hook until it's approved via `/hooks` — and it prints `hook: X Completed` even when it skipped it, so its output is never proof a hook ran; only an arriving event is (ADR-0006). Debugging notes: `docs/codex-hooks.md`.
- `hooks_install.rs` — merges 7 curl-command hooks into `~/.claude/settings.json` (backs up to `settings.json.bak-tokibean` first; idempotent by checking for the `:<port>/event` marker string in each event's hook array; `incomplete()` reports if any event or WSL distro is missing the marker, so the panel can re-surface the install button after an upgrade). On Windows it also syncs hooks into every WSL distro's existing `~/.claude` (via `\\wsl$`) — Claude Code in WSL reads the Linux-side settings, so Windows-only install is invisible to it. Mirrored networking uses `127.0.0.1`; NAT mode embeds `$(ip route show default | awk '{print $3}')` as host and additionally needs `bind: "0.0.0.0"` in the pet config. Uses curl commands rather than http-type hooks for Claude Code version compatibility.

**Frontend (`src/`)** — no framework, uses `window.__TAURI__` global (`withGlobalTauri: true`):

- `main.js` — listens for `pet-update`, invokes Tauri commands (`get_update`, `install_hooks`, `set_mode`, `get_config`, `set_config`, `connect_claude`, `panel_opened`), renders the usage panel, runs the canvas animation loop. Also owns: window drag-to-move (native `startDragging`, distinguished from a click by a 4px movement threshold), the expand/collapse panel animation, an 8-bit WebAudio beep on state changes, and dynamic skin loading (injects `skins/<name>.js` as a `<script>` tag when a non-default skin is configured).
- `pet.js` — the skin layer, deliberately isolated. All drawing lives here behind one interface: `window.PetRenderer.draw(ctx, canvas, state, warn, bubble, t, extra)` with states `idle | working | attention | done | limit`. `extra` is optional (`{sessions, workSecs, attnSecs, toolNote, celebrate, oops, bgCount, dragging, pat}`) — old skins that ignore it must keep working. Replacing this file swaps the skin; do not leak rendering logic into main.js.
- Skins live in `src/skins/*.js` (override `window.PetRenderer`, may reuse `window.PetKit`). The default skin in pet.js IS the original mascot "拱门·墩墩" (persimmon arch); `bean.js`/`tabby.js` are the other built-in skins. `src/skins/tribute.js`, if present, is a LOCAL-ONLY skin (gitignored) — never commit it or recreate its likeness in the repo.
- **Windows and macOS never resize the window** (`PREALLOC` in main.js): the collapsed window is pre-allocated to the full panel height (`FULL_H`, set in main.rs `setup`), and opening the panel only un-hides the already-allocated DOM — nothing moves, nothing resizes. This is load-bearing, not an optimization: any resize that moves the window's *top* edge makes the pet flash (newH − oldH) px up and snap back, because the webview keeps painting its OLD layout for one frame after the window geometry changes (Windows: DWM re-composites the stale frame; macOS: the WKWebView surface lags). Doing the move+resize atomically (`set_window_rect`) does **not** fix it — the stale frame is the webview's, not the window's. Only Linux still resizes (`setWindowLayout`). Consequences: the pet's saved position is the layout-independent **pet anchor** (`pet_anchor_y` = the pet canvas-top on screen, written by `savePetAnchor` and by the `Moved` backstop), because the full-height window's top-left depends on the up/below layout; `#panel` is `flex: none` + `max-height` (a flex column would otherwise shrink it inside the window and clip its top rows); and the click-through thread makes only the pet strip solid (`pet_at_top` picks which end of the window it's at), since the window is mostly transparent. Window position persists to config (throttled save on Moved, skipped within 1.2s of a programmatic resize, since panel toggle moves-then-resizes and that "move" isn't a user drag).
- `gen-icon.js` — zero-dependency Node script that hand-draws `app-icon.png` (1024×1024) pixel-by-pixel and PNG-encodes via built-in `zlib`, mirroring the default skin's pixel geometry. Run before `npm run tauri icon app-icon.png` if the mascot art changes.

## Constraints

- The default skin references the official pixel mascot and is for personal use only — if asked to prepare this project for public distribution, `pet.js` needs an original character and "Claude" must be dropped from the name (see README).
- Changing the port requires reinstalling hooks (the port is baked into the curl commands in settings.json and into `TOKIBEAN_PORT` in the installed Hermes plugin).
- Rust holds locks briefly and clones snapshots out; keep that pattern — the hook server, heartbeat, and click-through threads share every Mutex in `Shared`.
- OAuth/credential handling: tokens must never appear in a process's command-line args (always pipe via curl stdin) and never get logged. The pet uses only its own connected credential (`login.rs` → config) and refreshes it independently with backoff; it does **not** read or refresh other clients' credential stores (`.credentials.json`, WSL, Keychain, Credential Manager) — a standalone app shouldn't hang its reliability on another client's rotating token, and refreshing someone else's credential rate-limits the shared token endpoint / kicks that client offline.
